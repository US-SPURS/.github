# Contributing to US-SPURS

Thank you for your interest in contributing to the United States Department of Special Projects and Unified Response Services (US-SPURS) open source projects!

## Welcome!

We're excited that you're considering contributing to US-SPURS projects. This document provides guidelines for contributing to our repositories. By participating in this project, you agree to abide by our [Code of Conduct](CODE_OF_CONDUCT.md).

## How to Contribute

### Reporting Bugs

Before creating bug reports, please check the existing issues to avoid duplicates. When you create a bug report, include as many details as possible:

* **Use a clear and descriptive title** for the issue
* **Describe the exact steps to reproduce the problem** in as much detail as possible
* **Provide specific examples** to demonstrate the steps
* **Describe the behavior you observed** and explain why this is a problem
* **Explain the behavior you expected** instead
* **Include screenshots or animated GIFs** if relevant
* **Include details about your environment** (OS, browser version, etc.)

### Suggesting Enhancements

Enhancement suggestions are tracked as GitHub issues. When creating an enhancement suggestion, include:

* **Use a clear and descriptive title**
* **Provide a detailed description** of the suggested enhancement
* **Provide specific examples** to demonstrate the steps or use cases
* **Describe the current behavior** and explain the behavior you'd like to see
* **Explain why this enhancement would be useful**

### Pull Requests

1. Fork the repository and create your branch from `main`
2. If you've added code that should be tested, add tests
3. If you've changed APIs, update the documentation
4. Ensure the test suite passes
5. Make sure your code follows the existing style
6. Write a clear commit message
7. Include the relevant issue number if applicable

#### Pull Request Process

1. Update the README.md with details of changes to the interface, if applicable
2. Update any relevant documentation
3. The PR will be merged once you have the sign-off of at least one maintainer

### Coding Standards

* Follow the existing code style in the project
* Write clear, readable, and maintainable code
* Comment your code where necessary
* Write meaningful commit messages
* Keep pull requests focused on a single concern

## Development Setup

Each repository may have specific setup instructions in its README.md file. Generally:

1. Fork and clone the repository
2. Install dependencies as specified in the project documentation
3. Create a feature branch for your changes
4. Make your changes and test thoroughly
5. Submit a pull request

## Testing

* Write tests for new features and bug fixes
* Ensure all tests pass before submitting a pull request
* Follow the testing conventions used in the project

## Documentation

* Keep documentation up to date with code changes
* Use clear and concise language
* Include code examples where appropriate
* Update the README.md if you change functionality

## Community

* Be respectful and constructive in all interactions
* Help others when you can
* Follow our [Code of Conduct](CODE_OF_CONDUCT.md)

## Questions?

If you have questions about contributing, please open an issue or contact us at opensource@us-spurs.gov.
