# Governance

## Overview

The United States Department of Special Projects and Unified Response Services (US-SPURS) is an Independent Federal Agency (IFA) operating from Joint Force Base Freedom Island (formerly Alameda Naval Air Station).

This document outlines the governance structure for US-SPURS open source projects.

## Roles and Responsibilities

### Maintainers

Maintainers are responsible for:
* Reviewing and merging pull requests
* Triaging issues and bugs
* Setting project direction and roadmap
* Ensuring code quality and security
* Managing releases
* Enforcing the Code of Conduct

Maintainers are listed in the [MAINTAINERS.md](MAINTAINERS.md) file.

### Contributors

Contributors are community members who contribute to the project through:
* Code contributions
* Documentation improvements
* Bug reports and feature requests
* Community support

All contributors are expected to follow our [Code of Conduct](CODE_OF_CONDUCT.md) and [Contributing Guidelines](CONTRIBUTING.md).

## Decision-Making Process

### Consensus-Based

US-SPURS projects use a consensus-based decision-making model. Maintainers strive to reach agreement on significant changes through discussion and compromise.

### When Consensus Cannot Be Reached

If maintainers cannot reach consensus on a significant issue, the decision will be escalated to the US-SPURS Open Source Program Office for final determination.

## Communication Channels

* **GitHub Issues**: For bug reports, feature requests, and technical discussions
* **Pull Requests**: For code contributions and reviews
* **Email**: opensource@us-spurs.gov for general inquiries

## Code Review Process

All changes to US-SPURS repositories must be reviewed by at least one maintainer before merging. Reviews should focus on:

* Code quality and maintainability
* Security considerations
* Alignment with project goals
* Documentation completeness
* Test coverage

## Release Process

Release procedures are documented in individual project repositories. Generally:

1. Maintainers prepare release notes
2. Version numbers follow semantic versioning
3. Releases are tagged in Git
4. Release artifacts are published as appropriate
5. Security updates are prioritized

## Changes to Governance

This governance document may be updated by consensus of the maintainers. Significant changes should be communicated to the community with opportunity for feedback.

## Questions

For questions about governance, please contact opensource@us-spurs.gov.
